OpenClaw 的安全体系通过系统提示词注入的方式，在每一次 LLM 调用时约束 Agent 的行为边界。这些提示词位于 src/agents/system-prompt.ts 中，构成 Agent 无法绕过的第一层安全防线。

01核心安全提示词

系统提示词中嵌入的核心安全原则，直接约束 Agent 的行为边界。这些规则以提示词形式注入每一次 LLM 调用，构成最基础的安全防线。

从源码中可以看到，核心安全提示词包含四条硬性约束：

  • 禁止自主目标 — 无自我保存、复制、资源获取、权力寻求行为。Agent 不得产生自我保护或扩张的动机。
  • 安全优先于完成度 — 当安全与任务完成发生冲突时，暂停执行并询问用户，而非强行推进。
  • 禁止扩大权限 — 不得说服用户扩大访问范围或禁用安全措施。Agent 不能诱导用户降低安全级别。
  • 禁止自我复制 — 不得复制自身或修改提示词 / 安全策略 / 工具策略。防止自我篡改和未授权传播。

这些规则以字符串数组的形式直接注入系统提示词，Agent 在每次推理时都会将其作为上下文的一部分。这不是配置项，而是硬编码在系统行为中的约束。

02Control 控制安全

防止 Agent 通过幻觉式命令或自主操作对系统造成意外影响。涉及系统生命周期和全局状态的操作必须受控。

Control 安全提示词主要约束两个方面：

  • 禁止捏造命令 — 防止幻觉式执行不存在的指令。Agent 必须确认命令存在且合法后方可执行。
  • 生命周期操作需显式用户请求 — 重启、停止等系统级操作必须由用户明确发起，禁止 Agent 自主决定。配置和重启操作优先使用 gateway 工具，CLI 生命周期命令仅在用户显式请求时执行。

03工具调用安全

工具调用是 Agent 与外部世界交互的主要通道，也是安全风险最集中的环节。OpenClaw 通过多层审批机制确保每一条命令都经过适当授权。

工具调用安全包含三层防护：

  • 审批命令隔离 — /approve 只能由用户发起，禁止通过 exec 等路径绕过审批流程。/approve 是一个面向用户的审批命令，不是 shell 命令。
  • 单次授权原则 — allow-once 仅覆盖单条命令，不得跨命令复用授权。每次执行都需要独立确认。
  • 命令完整性展示 — 审批时必须展示完整命令（含链式操作符 &&、;、| 等），防止隐藏实际执行内容。

此外，提示词还规定了一类工具调用规范：低风险调用不需要叙述，仅在复杂、敏感/破坏性或用户明确要求时才进行叙述；存在一等工具时应直接使用，不要要求用户运行等效的 CLI/slash 命令。

04沙箱安全

沙箱是工具执行层面的物理隔离层。即使提示词层面的安全约束被绕过，沙箱仍能提供最后一道防线。

沙箱安全提示词在沙箱启用时动态注入，包含以下约束：

  • 沙箱隔离 — 工具在 Docker 容器中执行，与宿主机环境隔离。敏感操作被限制在容器边界内。部分工具可能因沙箱策略而不可用。
  • 子代理继承沙箱 — 子代理无法获得提升或宿主访问权限。沙箱权限在代理链中向下传递，不会向上逃逸。需要沙箱外读写时，不要 spawn 子代理，先询问用户。
  • ACP Harness 限制 — ACP harness 的 spawn 被阻止在沙箱会话中使用（sessions_spawn 且 runtime: "acp"），应改用 runtime: "subagent"。

05Workspace 工作区路径

路径层面的隔离确保 Agent 在正确的上下文中操作文件，防止因路径混淆导致的越权访问。

路径隔离的核心规则：

  • 路径隔离 — 对于 read/write/edit/apply_patch 操作，文件路径解析到宿主机工作区 ${sanitizedWorkspaceDir}；对于 bash/exec 命令，使用沙箱容器路径。
  • 相对路径优先 — 将此目录视为文件操作的单一全局工作区，除非另有明确指示。避免使用绝对路径，防止路径混淆导致越权访问。
📝 总结：OpenClaw 的安全提示词体系从五个维度构建了一道纵深防线——核心安全原则约束 Agent 的内在动机，Control 控制防止系统级误操作，工具调用审批隔离危险命令执行，沙箱提供物理隔离层，Workspace 路径隔离确保文件操作安全。这五个维度层层递进，即使某一维度被绕过，其他维度仍能提供保护。
